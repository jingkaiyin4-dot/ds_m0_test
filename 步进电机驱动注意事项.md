# 步进电机驱动注意事项

## 1. 本项目最终验证通过的接线与资源

- 电机1: `UART_bj1`
- 外设实例: `UART3`
- 引脚: `PA14(TX)` / `PA13(RX)`
- 地址: `1`

- 电机2: `UART_bj2`
- 外设实例: `UART0`
- 引脚: `PA0(TX)` / `PA1(RX)`
- 地址: `2`

- 串口参数统一为: `115200 8N1`
- 两个电机是分别接在两个独立串口上，不是同一总线多地址混挂

## 2. 这次调通后确认的核心结论

1. 同一套驱动协议在 MSPM0 上可用，但前提是发送字节时序不能乱改。
2. `ZdtStepper_SendBytes()` 用最初那个“等 TX FIFO 不满 -> 写数据 -> 最后等 UART busy 结束”的版本可以正常工作。
3. 之前尝试模仿 F4 改成“每个字节发送后等待 TX empty”这一版，虽然逻辑看起来更像 F4，但在这个工程里会把发送行为改坏，导致电机不转。
4. 电机1和电机2能分别转动，说明协议本身没有问题，关键在于串口实例、地址和发送实现要同时匹配。

### 电机2

同步启动命令:

## 4. 调试时最容易踩的坑

### 4.1 不要随便改发送函数

这次最关键的经验就是:

- `app/ZdtStepper.c` 里的 `ZdtStepper_SendBytes()` 一旦改成别的等待方式，电机可能马上不转
- 即使串口抓包看起来“字节内容一样”，电机也可能因为发送节奏不同而不响应

当前验证可用的发送逻辑特征:

- 发送前等待 `TXFIFO` 非满
- 逐字节写入
- 帧结束后等待 `UART busy` 结束

### 4.2 地址和串口必须同时对应

这里不是简单“换个地址就行”，而是要同时满足:

- 电机1 -> `BJ1/UART3` -> 地址 `1`
- 电机2 -> `BJ2/UART0` -> 地址 `2`

如果只改地址，不改对应串口，或者硬件上接的是另一个电机，就会误判为协议错误。

### 4.3 两个电机是独立串口，不是同一串口多地址总线

这次必须记住:

- 现在项目里是两个物理电机分别接两个 UART
- 区分电机靠的是“串口 + 地址”联合确定
- 不能只凭地址判断目标电机

### 4.4 上电后要给驱动器初始化时间

测试中保留了上电后 `500ms` 延时，能避免驱动器还没稳定就收命令。

推荐流程:

```text
系统初始化 -> 延时500ms -> 发速度命令 -> 延时10ms -> 发同步启动
```

### 4.5 速度命令后必须补同步启动

只发 `F6` 速度命令不够，还要继续发:

```text
00 FF 66 6B
```

否则电机可能不会真正动作。

## 5. 本次最终可用的软件结构

关键文件:

- `app/ZdtStepper.c`: 步进电机底层发送和协议封装
- `app/ZdtStepper.h`: 驱动接口声明
- `app/ZdtStepperTest.c`: 当前双电机测试逻辑
- `main.c`: 主循环切到 `ZdtStepperTest`
- `mpu6050-oled-hardware-i2c.syscfg`: UART 引脚分配
- `Debug/ti_msp_dl_config.h`: SysConfig 生成后的 UART 定义

## 6. 后续如果再出问题，优先检查顺序

1. 先看 `syscfg` 里的串口引脚有没有被别人改掉
2. 再看 `ZdtStepper_SendBytes()` 是否被改过
3. 再确认测试代码到底在发 `BJ1` 还是 `BJ2`
4. 再确认地址是不是 `1/2` 对应正确
5. 最后再抓串口波形，不要一开始就怀疑协议本身

## 7. 建议保留的原则

- 已验证能转的底层发送函数，不要再轻易“优化”
- 改一个变量只改一个维度，不要同时改串口、地址、时序
- 先让单电机稳定，再扩展双电机联动
- 每次调通后立刻记录“串口 / 地址 / 引脚 / 命令帧 / 延时”

## 8. 本次复盘一句话总结

这次不是协议本身难，而是“硬件串口映射 + 电机地址 + UART发送实现”三个条件必须同时完全对上，任何一个地方改乱了，串口看着正常，电机也照样不转。
